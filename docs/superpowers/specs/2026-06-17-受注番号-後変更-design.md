# 受注番号の後変更（リネーム）機能 — 設計仕様

作成日：2026-06-17 / 対象：`案件管理アプリ/index.html`（単一HTMLファイル）

## 1. 目的・背景

受注番号（案件番号。例 `T26045`／単独請求書 `inv_T26xxx`）は登録後ロックされ変更できない。あとから番号を訂正できるようにする。受注番号は `ws_${num}` / `inv_${num}` の **localStorage キーそのもの**であり、projects_saved・wsData・invData・deleted_ws/inv・Googleスプレッドシート同期（order/child/invoice、キー列「受注No」）の**主キー**。したがって「変更」＝キーのリネームで、全参照の整合更新が必要。

## 2. 決定事項（ユーザー確認済み 2026-06-17）

1. **変更場所**
   - 案件（ワークシート）：**データ管理画面に「受注番号の変更」ツールを新設**。一覧から選んで変更。紐づく請求書（同じ番号の `inv_`）も自動で同番号に反映。
   - 単独請求書（ワークシート無し）：**請求書編集画面**のロックされた受注番号欄に「✏️変更」を付け、押すと編集可。同じ変更ロジックを呼ぶ。
   - データ管理の一覧には**案件と単独請求書の両方**を出す（番号変更はここで完結できる）。
2. **重複時**：警告を出し、**どちらにその番号を持たせるか**ユーザーが選ぶ。**負けた方（番号を譲る側）は番号が「0000」になる**（再採番の目印）。「0000」が既に使用中なら 0001, 0002… と空きを使う。
3. **請求済み/入金済み**：変更**可**。ただし `status.billed` または `status.paid` の場合は「客に出した番号が変わります。本当に変更しますか？」の確認を1回出す。

## 3. UI 設計

### 3.1 データ管理画面「受注番号の変更」セクション
- 既存のデータ管理画面（バックアップ/復元/リセットがある画面）に新セクションを追加。
- 一覧：番号変更の対象を「案件（ワークシート）」＋「単独請求書」で表示。各行 = `[受注番号] [件名/取引先] [✏️変更]`。
  - データ源：`getSavedProjects()`（案件）＋ `inv_` キー走査で projects_saved に無い単独請求書。`deleted_ws`/`deleted_inv` は除外。
- 「✏️変更」押下 → 接頭辞ボタン(T/E/P)＋数字入力の小フォーム（既存の登録UIの作法に合わせる）→「変更する」。

### 3.2 請求書編集画面（単独請求書）
- 現在 `_setInvOrderField(num, false)` でロック表示している受注番号欄（`inv-order-readonly` 表示 / `inv-order-edit` 非表示）の横に「✏️変更」ボタンを追加。
- 押すと `inv-order-edit`（接頭辞ボタン＋ `inv-order-num`）を表示し編集可に。「変更する」で確定 → 共通リネーム処理へ。
- 案件に紐づく請求書（ws_ が存在する番号）はここでは変更させない（データ管理画面の案件側で行う）。＝この欄の✏️は「単独請求書」のときのみ表示。

## 4. 中核：リネーム処理 `renameOrderNum(oldNum, newNum)`

新規追加する単一の純粋関数。**newNum は呼び出し前に空き（衝突なし）であることを前提**（衝突解決は呼び出し側＝UIで行う）。以下を順に実行：

1. **正規化・前提チェック**
   - newNum を既存ルールでサニタイズ（英数字とハイフンのみ、記号除去、大文字化等は既存採番に合わせる）。空・不正なら中止。
   - `oldNum === newNum` なら何もしない（no-op）。
2. **localStorage キー移行**
   - `ws_${oldNum}` があれば → `ws_${newNum}` に setItem、`ws_${oldNum}` を removeItem。
   - `inv_${oldNum}` があれば → `inv_${newNum}` に setItem、`inv_${oldNum}` を removeItem。
3. **projects_saved**：`num === oldNum` の要素を探し、`num = newNum`、`_updatedAt = now`。無ければスキップ（単独請求書）。保存。
4. **メモリ**：`wsData[oldNum]` があれば `wsData[newNum] = （num更新）`、`delete wsData[oldNum]`。`invData` も同様（あれば）。
5. **削除リスト**：`deleted_ws`/`deleted_inv` に oldNum があれば newNum に置換（通常は無いが防御）。
6. **Googleスプレッドシート同期**（二重行防止）
   - 旧番号を消す：`pushKind('order',[{受注No:oldNum,_deleted:true,_updatedAt:now}])`、`'child'` も同様、請求書があれば `'invoice'` も同様。
   - 新番号で作り直す：`pushKind('order',[{受注No:newNum, _project:wsData[newNum], _updatedAt:now}])`、`'child'` は ws データに `受注No:newNum` を付与、請求書があれば `'invoice'` に新番号付与。
   - ※ペイロード形状は既存の `saveWS`/`saveInvStatus`/削除処理の `pushKind` 呼び出しと**完全に一致**させる（実装時に該当箇所を確認してコピー）。同期未接続/オフラインなら `pending` キューに積まれる既存仕組みに乗る。
7. **UI 反映**
   - 開いていれば `ws-detail-num` / `inv-detail-num` の textContent を更新。
   - 再描画：ホームのスケジュール、売上表、書類/請求書リスト、データ管理の一覧、カードの `data-wsnum`/onclick。既存の再描画フック（renderHomeCal 等）を呼ぶ。

戻り値：成功可否（必要なら変更内容）。

## 5. 衝突解決フロー（UI 層・呼び出し側）

案件/請求書 A を newNum=X に変更しようとした時：

1. **形式チェック** → 不正なら中止。
2. **請求済み確認**：A が billed/paid なら確認ダイアログ。NO なら中止。
3. **衝突チェック**：`_orderNumExists(X)` が真かつ X が A 自身でない → 衝突。
   - 衝突相手 B の情報（番号・件名）を取得し、警告ダイアログ：「`X` は『B件名』が使っています。どちらに `X` を付けますか？（負けた方は番号が 0000 になります）」。3択：
     - **［こちら(A)を優先］** → 先に `renameOrderNum(B, _freeZeroNum())`（B を 0000 系へ退避し X を空ける）→ 続けて `renameOrderNum(A, X)`。結果：A=X、B=0000。
     - **［相手(B)を優先］** → B は X のまま。`renameOrderNum(A, _freeZeroNum())` で A を 0000 系へ。結果：B=X（不変）、A=0000。
     - **［キャンセル］** → 何もしない（A は元の番号のまま）。
4. 衝突なし → 請求済み確認の後 `renameOrderNum(A, X)`。

補助関数 `_freeZeroNum()`：`0000` から始め `_orderNumExists` で空きを探して返す（`0000`,`0001`,…）。

「0000」になった案件は一覧・カードで目立つ表示（例：`0000` を赤＋「要・番号再設定」）にして、ユーザーが後で正しい番号に振り直せるようにする。

## 6. 影響箇所（調査済み・実装時に再確認）

- 保存/読込：`saveWS`／`loadWS`（`ws_${num}`）、`saveInvStatus`／`loadInvStatus`（`inv_${num}`）。
- 一覧/採番：`getSavedProjects`、`getNextOrderNum`、`_orderNumExists`（衝突チェックに流用）。
- 同期：`pushKind`、`SYNC_KINDS`（order/child/invoice、キー「受注No」）、削除フラグ送信の既存箇所。
- 参照：`wsData`、`invData`、`getDeletedWS`/`deleted_ws`、`deleted_inv`、`_collectSalesItems`、`_collectDocItems`。
- 画面：`ws-detail-num`、`inv-detail-num`、`inv-order-edit`/`inv-order-num`/`inv-order-readonly`、`_setInvOrderField`、データ管理画面、各カード `data-wsnum`。

## 7. エッジケース

- newNum が空/記号のみ/自分と同じ → 何もしない or 中止。
- A が案件＋請求書の両方を持つ → 同じ num なので両方一括でリネーム（キー共有のため自動）。
- A が単独請求書（projects_saved 無し）→ inv_ のみリネーム、projects_saved はスキップ。
- 衝突相手 B も請求済み → B を 0000 にする前に追加確認は出さない（A 側で既に意思確認済み・B はあくまで退避）。ただし「Bが0000になります」と警告文に明記。
- `0000` 自身を新番号に手入力 → 通常運用では使わない想定だが、形式上は許容（特別扱いしない）。
- 同期オフライン中の変更 → pending キューに積まれ、復帰時に旧削除＋新作成が送られる（既存挙動）。

## 8. テスト計画（実装後）

- `node --check` 相当（script 抽出構文チェック）。
- ブラウザ実機（同期は `window._gasUrlOne=()=>''` で無効化し本番シート保護）：
  1. 案件のみ（請求書無し）の番号変更：ws_/projects/wsData/カード/スケジュールが新番号に。旧キー消滅。
  2. 案件＋紐づく請求書：両方が同番号で変わる。売上表・請求書リストに新番号で1件、旧番号無し。
  3. 単独請求書：請求書編集画面の✏️から変更。inv_ リネーム、売上表反映。
  4. 衝突：既存番号へ変更→警告（3択）。「こちらを優先」で相手が 0000・自分が新番号／「相手を優先」で自分が 0000・相手は不変／「キャンセル」で無変更。各分岐でキー整合を確認。
  5. 0000 が既存 → 0001 が使われる。
  6. 請求済み案件 → 確認ダイアログが出る。キャンセルで無変更。
  7. 形式：記号入り/空/自分と同じ番号の扱い。
  8. UI 4パターン（スマホ/PC×文字100%/150%）でデータ管理の新セクション・請求書欄が崩れない。
  9. 同期ペイロード：`_gasUrlOne` を一時モックして pushKind が「旧=削除フラグ」「新=データ」で正しい形で呼ばれるかを確認（本番送信はしない）。

## 9. やらないこと（YAGNI）

- 一括/複数同時リネーム、リネーム履歴・取り消し（undo）は作らない。
- 接頭辞ごとの連番の自動振り直しはしない（手入力した新番号をそのまま使う）。
- 「0000」の自動再採番はしない（目印を出すだけ。ユーザーが手で振り直す）。
