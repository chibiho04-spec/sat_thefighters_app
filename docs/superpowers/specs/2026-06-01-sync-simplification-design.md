# 同期のシンプル化 — 設計ドキュメント

作成日: 2026-06-01
対象アプリ: Sat the Fighters（案件管理アプリ / 単一HTML構成 `index.html`）
デプロイ: GitHub Pages（`https://chibiho04-spec.github.io/sat_thefighters_app/`）
前提ドキュメント: `docs/superpowers/specs/2026-05-31-google-sheets-sync-design.md`（現行as-built）

## 背景・目的

現行の同期は **8個の別々のスプレッドシート＋8個の Apps Script＋8個のURL** で構成され、

1. 端末ごとに URL を8個＋合言葉を貼る必要があり設定が大変
2. 種類ごとにマージ方式がバラバラ（請求書=LWW / 機材=全置換union＋墓標 / スタッフ=無条件上書き）でデータ消失事故が起きる
3. 仕組みを頭の中で追えない
4. 種類や項目を足すたびコピペ関数を増やす必要があり保守が大変

という4点すべてが「複雑さ」の原因になっている。本設計は次の3本柱で刷新する。

- **バックエンドを1個に統合**：1スプレッドシート（タブで種類を分ける）＋1 Apps Script＋1 URL＋1合言葉。
- **マージ規則を1つに統一**：全タブ共通の「サーバ側 Last-Writer-Wins（更新日時で後勝ち）＋削除フラグ」。
- **アプリ側コードを1エンジンに統合**：種類ごとの設定を登録表に持たせ、共通の push/pull/syncAll で回す。

**読みやすさは犠牲にしない**：案件など現状で人が読める列構成のタブは、その列構成を残す（JSONに潰さない）。

## 非対象（YAGNI）

- リアルタイム同期、項目単位マージ、ユーザー別アクセス制御。
- 認証方式の変更（合言葉トークン方式は維持）。
- 既存データの破棄。移行は新シートへ全件シードで行い、旧8シートは保険として残置可。

## ① 全体アーキテクチャ

```
各端末（アプリ / localStorage）
   ⇅ HTTPS（GET=取り込み / POST=送信、Content-Type: text/plain でCORSプリフライト回避）
1個の Apps Script Web App（パラメータ kind で行き先タブを振り分け）
   ⇅ SpreadsheetApp
1個のスプレッドシート（タブ：案件 / 子 / 取引先 / 商品 / メディア / 機材 / 請求書 / スタッフ）
```

- 通信は現行同様すべて `Content-Type: text/plain`。GET は毎回 `_ts=Date.now()`＋`cache:'no-store'`。
- 端末が保持する同期設定は **URL1個（`gas_url`）＋合言葉1個（`gas_token`）** のみ。
- 合言葉はURLクエリに `?token=` で自動付与（現行の fetch ラッパーを流用）。

## ② データモデル（タブごとに列は自由・共通3列だけ必須）

各タブは **現状の列構成をそのまま維持**する。ただし全タブに次の3つの列を必ず持たせる（ヘッダ名で識別）。

| 共通必須列（ヘッダ名） | 役割 |
|---|---|
| キー列 | 1レコードを一意に識別。タブごとに名前が違ってよい（後述） |
| `更新日時` | ISO文字列。後勝ち判定に使う |
| `削除フラグ` | 空 or `1`。`1`なら論理削除 |

### タブ別の構成

| タブ | キー列ヘッダ | 中身の列 | 形式 |
|------|------------|----------|------|
| 案件 | `受注No` | 受注日・受注担当・取引先・件名・撮影日程・各クルー等（読める列） | 読める列 ＋ 共通3列 |
| 子（ワークシート詳細） | `受注No` | 詳細データ列（現状の子シート構成） | 読める列/JSON ＋ 共通3列 |
| 取引先 | 取引先コード（実装時に現状ヘッダを確認） | 取引先名・コード等 | 読める列 ＋ 共通3列 |
| 商品 | 商品コード | 商品名・単価等 | 読める列 ＋ 共通3列 |
| メディア | メディアID（実装時に現状ヘッダを確認） | メディア項目 | 読める列 ＋ 共通3列 |
| 機材 | `id`（em001等） | カテゴリ・名称・コード・数量・単価 | 読める列 ＋ 共通3列 |
| 請求書 | `受注No` | `データ`（JSON丸ごと） | JSON方式（現状維持） |
| スタッフ | `キー`（kawasaki等） | `データ`（JSON丸ごと） | JSON方式（現状維持） |

- 「読める列」タブと「JSON」タブが混在してよい。**共通なのはキー・更新日時・削除フラグの3列だけ**。
- 削除は行を消さず `削除フラグ=1`＋`更新日時=今`。これにより現行の削除管理キー
  `deleted_ws` / `deleted_inv` / `equip_deleted` を**廃止**できる。
- 機材の `action:'sync'`（全置換）＋union＋墓標の特殊処理を**廃止**し、機材も1個1行の upsert に統一する。

#### 案件タブへの列追加（現状からの差分）

現状の `projectToOrderRow` は `受注No` ＋ 各項目列のみで、`更新日時`/`削除フラグ` 列を持たない
（削除は `action:'delete'` ＋ ローカル `deleted_ws` で処理）。新モデルでは案件タブにも
`更新日時`・`削除フラグ` の2列を追加し、削除を共通の論理削除に寄せる。

## ③ サーバ側 Apps Script（共通1本）

1個の Web App。GET/POST 両方で `e.parameter.token` を検査し、`SYNC_SECRET` と不一致なら拒否。

### ルーティング

- `kind`（例 `order`/`child`/`client`/`product`/`media`/`equip`/`invoice`/`staff`）でタブを選ぶ。
- `kind → タブ名` の対応表をスクリプト先頭に持つ。未知の `kind` はエラー応答。

### GET `?action=list&kind=xxx`

- 指定タブの2行目以降を「ヘッダ名→値」のオブジェクト配列で返す（`{ ok:true, rows:[...] }`）。
- キーが空の行はスキップ。削除フラグ行も**そのまま返す**（削除の伝播はクライアントが判断）。

### POST `{action:'upsert', kind:'xxx', rows:[...]}`

- `LockService.getScriptLock()` で直列化（同時実行の重複行を防止）。
- 各行について、キー列の値で既存行を検索。
  - **既存あり**：シート行の `更新日時` と受信行の `更新日時` を**文字列比較**し、
    **受信が新しいときだけ上書き**（古い/同値なら無視）。これがサーバ側LWWの肝。
  - **既存なし**：末尾に追加。
- ヘッダに無いキーは無視。受信行に無い列は空のまま。
- 重複キー行があれば更新日時が新しい1行を残して他は削除（現行 staff Code.gs のdedupを踏襲）。
- 応答：`{ ok:true, updated, added, skipped, removedDup }`。

### 補助

- `ensureHeaders(kind)`：タブが無ければ作成し、そのタブ既定のヘッダ行を書き込む（初回シードを楽にする）。
- `cleanupDuplicates(kind)`：手動実行用のdedup。

> **設計上の効果**：サーバが更新日時を見て古い上書きを拒否するため、ある端末が未編集の初期値を
> 送っても他端末の新しい編集を壊さない。アプリ側の「未編集は送らない」等の小細工が不要になる。

## ④ アプリ側エンジン（登録表＋共通処理）

### 登録表（kind ごとの設定を1か所に集約）

各 kind について次を持つオブジェクトの配列／マップを定義する。

```
SYNC_KINDS = {
  order:   { tab:'案件',     keyField:'受注No', loadLocal, saveLocal, toRow, fromRow },
  child:   { tab:'子',       keyField:'受注No', ... },
  client:  { tab:'取引先',   keyField:'取引先コード', ... },
  product: { tab:'商品',     keyField:'商品コード',   ... },
  media:   { tab:'メディア', keyField:'メディアID',   ... },
  equip:   { tab:'機材',     keyField:'id',     ... },
  invoice: { tab:'請求書',   keyField:'受注No', ... },  // toRow は JSON 丸ごと
  staff:   { tab:'スタッフ', keyField:'キー',   ... },  // toRow は JSON 丸ごと
}
```

- `loadLocal()` … その種類の全レコードを localStorage から配列で返す。
- `saveLocal(list)` … マージ結果を localStorage に書き戻す。
- `toRow(record)` … レコード→シート1行オブジェクト（キー列・各列・`更新日時`・`削除フラグ`）。
- `fromRow(row)` … シート1行→レコード（`更新日時`・`削除フラグ`含む）。

種類ごとに違うのは列構成だけなので、差分は `toRow`/`fromRow`/`keyField`/`tab` に閉じる。

### 共通関数

- `pushKind(kind, records?)`：`records` 未指定なら未送信分。`toRow` で行化し
  `POST {action:'upsert', kind, rows}`。成功で `sync_pending` から除去、失敗で追加。
- `pullKind(kind)`：`GET ?action=list&kind` → 各行を `fromRow` → ローカルとLWWマージ（更新日時で後勝ち、
  `削除フラグ=1` かつシートが新しければローカル削除）→ `saveLocal` → 再描画フック。
- `syncAll()`：全 kind を pull → その後 push（手動「全部同期」ボタン）。**pull優先**で、
  自端末の古い値が新しい取り込みを壊さない順序にする。

### タイミング（現行を踏襲）

- 起動時（約1.5秒後）に各 kind を pull。
- 起動の取り込みが落ち着いた頃（約4秒後）に `sync_pending` を再送。
- 保存時に該当レコードをデバウンス（約1〜1.2秒）でまとめて `pushKind`。
- 手動「🔄 全部同期」ボタン（データ管理画面）。

### 設定UIの簡素化

- 同期URL入力欄を **8個 → 1個（`gas_url`）** に統合。合言葉欄は1個（現状どおり）。
- 旧 `gas_url_child` 等のキーは読み取りに使わない（移行後は不要）。後方互換のため即削除はしない。

## ⑤ 移行手順（データは消えない）

各端末はローカルに全データの控えを持つため、空の新バックエンドへ丸ごと移せる。

1. 新しい空のスプレッドシートを1個作成。
2. 新しい Apps Script に共通 `Code.gs` を貼り、`SYNC_SECRET` を既存と同じ合言葉に設定。ウェブアプリとしてデプロイ → URL1個を取得。
3. アプリのデータ管理→設定で「同期URL」にそのURL1個を入力・保存。
4. **1台のPCで「🚀 新シートへ全部送信（初回シード）」を1回押す** → `ensureHeaders` で全タブが作られ、各 kind の全レコードが書き込まれる。
5. 他端末はURL1個を入れて起動 → pull で全タブ取り込み。
6. 旧8シートは保険として残置。問題なければ後日削除。

## エラー処理（止まらない設計・現行踏襲）

- URL未設定：同期オフでローカル動作。
- 送信失敗：`sync_pending` に kind＋キーだけ積み、後で再送（中身は最新localStorageから作り直す）。
- 取り込み失敗：ローカル起動、ログのみ、後で再試行。
- 不正な応答（JSON以外等）：握りつぶしてローカル維持。
- 例外はログのみ（同期不調でアプリ本体を壊さない）。

## セキュリティ（内部ツール相応・現行踏襲）

- `script.google.com` 宛リクエストに fetch ラッパーが `?token=<合言葉>` を自動付与。
- GAS 側で GET/POST 両方の `e.parameter.token` を検査し不一致なら拒否。
- 合言葉はアプリ本体に書かない。各端末で1回入力し `gas_token`（その端末のみ）に保存。
- 既知の制約：URLと合言葉を知っていれば誰でも読み書き可能な簡易方式。社内限定運用が前提。

## 対象ファイル

- `index.html`（アプリ側：統合同期エンジン、登録表、設定UIの1URL化、各保存/削除フック、初回シードボタン）。
- `docs/superpowers/unified-sheet-Code.gs`（新規：共通 Apps Script 本体。リポジトリに置きユーザーが貼り付け）。
- `docs/superpowers/unified-sheet-setup.md`（新規：初心者向けセットアップ＆移行手順）。
- 既存の per-type 同期関数・3つの削除管理キー・機材 union/墓標処理は撤去対象。

## 成功基準

- 端末の同期設定が **URL1個＋合言葉1個** だけで済む。
- 全タブが **同一のLWW規則**で動き、未編集端末の送信が他端末の編集を壊さない（サーバ側で保証）。
- 案件など読める列のタブは Google シート上で**人が読める列のまま**。
- 削除が全タブ共通の論理削除で伝播し、`deleted_*` 系の補助キー無しで成立する。
- 1台で「全部送信」を押すと新シートに全データがシードされ、他端末が起動時に取り込める。
- 同期オフ（URL未設定）でも従来どおりローカル動作する。
- ネット不調でもアプリが止まらない（未送信は後で再送）。
