## 全体像

各社員のブラウザ（GitHub Pages のアプリ `index.html`）は、業務データを localStorage に保持しつつ、
**種類ごとに分かれた 8 個の Google Apps Script Web App** と双方向同期する。各 Web App は
オーナーの Google アカウント上の専用スプレッドシートに読み書きする。

```
各社員のブラウザ（index.html / localStorage）
   ⇅ HTTPS（GET=取り込み / POST=送信、Content-Type: text/plain でCORSプリフライト回避）
8 個の Google Apps Script Web App（種類別）
   ⇅ SpreadsheetApp
各種スプレッドシート（案件 / 子 / 取引先 / 商品 / メディア / 機材 / 請求書 / 運転記録）
```

- リアルタイム同期はしない（起動時取り込み＋保存時自動送信のハイブリッド）。
- 通信はすべて `Content-Type: text/plain`（CORSプリフライト回避）。GET は毎回 `_ts=Date.now()` を付け
  `cache:'no-store'` で取得（スマホの古いキャッシュ対策）。

## 8 個の Web App と localStorage キー

| 種類 | URL キー（localStorage） | 主なアクション | 備考 |
|------|--------------------------|----------------|------|
| 案件マスタ（親） | `gas_url` | `upsert` / `delete` | 全員共通。起動時に固定値を強制設定 |
| ワークシート詳細（子） | `gas_url_child` | `upsert` | 親に存在する案件にのみ詳細を補完 |
| 取引先マスタ | `gas_url_client` | `upsert` | |
| 商品マスタ | `gas_url_product` | `upsert` | |
| メディアリスト | `gas_url_media` | `upsert` | |
| 機材マスタ | `gas_url_equip` | `sync`（全置換） | id 単位の和集合マージ＋墓標で同時編集に対応 |
| 請求書 | `gas_url_invoice` | `upsert`（list取得） | 1案件1行・JSON丸ごと。更新日時で後勝ち |
| 運転記録 | `gas_url_driving` | `upsert` | |

### 同期に使うその他の localStorage キー

| キー | 用途 |
|------|------|
| `gas_token` | 同期の合言葉（端末ごとに1回入力・コードには未保存）。送信URLに `?token=` で自動付与 |
| `sync_pending` | 送信失敗の再送キュー（「何を送るか」だけを記録） |
| `deleted_ws` | 削除済みワークシート番号（案件削除の伝播） |
| `deleted_inv` | 削除済み請求書番号（請求書削除の伝播） |
| `equip_deleted` | 削除済み機材 id の墓標（機材削除の伝播） |

## 反映タイミング（ハイブリッド）

- **起動時（約1.5秒後）に取り込み**：URL設定済みの各シートを GET → ローカルへマージ → 再描画。
- **起動の取り込みが落ち着いた頃（約4秒後）に再送**：`sync_pending` に溜まった未送信分を自動再送。
- **保存時に自動送信**：各保存処理が当該レコードをデバウンス（約1〜1.2秒）でまとめて POST。
  - 案件保存：`sendOrderToSheet`（親）＋ `sendWorksheetToChild`（子）
  - 請求書保存：`scheduleInvoicePush` → `sendInvoiceToSheet`
  - 機材編集：`scheduleEquipPush` → `sendEquipMasterToSheet`
- **手動同期ボタン**：各セクションの「取り込み」「同期」「いま再送する」。

## マージ（競合解決）の方式

種類ごとに方式が異なる（実装の実態）。

### 請求書（最も整っている：Last-Writer-Wins）
- 各行に `更新日時`（`_updatedAt` の ISO 文字列）を持ち、取り込み時にローカルとシートの更新日時を比較し
  **新しい方を採用**。
- 削除は `削除フラグ=1` ＋ `更新日時=今`。取り込み時、ローカルに該当があり「シートの削除がローカルより新しい」
  場合はローカルから削除し `deleted_inv` に積む。
  → 「削除したのに古い編集で復活」「編集したのに古い削除で消える」を両方防ぐ。

### 案件（親＋子）
- 親シートは `upsert`（受注Noキー）。削除は `action:'delete'` で番号を渡し、ローカルは `deleted_ws` で管理。
- 子シートは「詳細データの補完」専用。親に存在する案件にのみ適用し、予約番号・削除済み・親に無い番号で
  勝手に新規ワークシートを作らない。

### 機材マスタ（全置換のため特別対応）
- 送信は `action:'sync'`（シート全体を置換）。古い一覧のまま送ると他端末の追加を消すため、
  **送信前にシートを取得し id 単位で和集合（union）マージ**してから送る（重複はローカル優先）。
- 取り込み時もシートとローカルを和集合マージ（重複はシート優先）。
- 削除は `equip_deleted`（墓標）に id を積み、マージ時に常に除外して伝播。
- **既知の限界**：墓標はサーバ側に列を持たないため、別端末が削除を受け取る前にその機材を編集すると、
  まれに復活することがある（再削除で消える）。完全化にはサーバ側の削除フラグ列が必要（今回は不採用）。

## 送信失敗時の再送キュー（`sync_pending`）

ネット不調などで送信に失敗したら、その「**種類（kind）＋番号（num）**」だけを `sync_pending` に積む
（中身そのものは保存しない）。起動時・手動ボタンでまとめて再送し、**再送時に最新の localStorage から
作り直す**ため古い内容で上書きしない。

- 主な関数：`pendingAdd / pendingClear / getSyncPending / retrySyncPending / updatePendingDisplay`。
- 各送信関数に「成功で除去・失敗で追加」のフックを実装（案件親/子・請求書/請求書削除・案件削除・機材）。
- 同じ対象は最新1件だけ保持（重複排除）。`_retryingPending` で多重実行を防止。
- データ管理画面に「未送信◯件」表示と「🔁 いま再送する」ボタン。

## セキュリティ（内部ツール相応）

- 送信先 `script.google.com` 宛のリクエストに、`fetch` ラッパーが自動で `?token=<合言葉>` を付与。
- GAS 側は `e.parameter.token` を GET/POST 両方で検査し、一致しなければ拒否（`Code.gs` ガード）。
- 合言葉は**アプリ本体に書かない**。各端末で1回入力し `gas_token`（その端末のみ）に保存。
- 既知の制約：URLと合言葉を知っていれば誰でも読み書き可能な簡易方式。社内限定運用が前提。
- 詳細手順：`docs/superpowers/google-sheets-token-setup.md`。

## エラー処理（止まらない設計）

- URL未設定：その種類は同期オフ。アプリは従来どおりローカル動作。
- 送信失敗：`sync_pending` に積み、後で自動再送。UIはブロックしない。
- 取り込み失敗：ローカルで起動、ログのみ。後で再試行。
- 不正な応答（JSONでない等）：握りつぶしてローカル維持。
- 例外はログのみ（同期不調でアプリ本体を壊さない）。

## アカウント移行

- 方法1：スプレッドシート＆Apps Script の所有権を会社アカウントへ移譲。
- 方法2：会社アカウントで新スプレッドシート＋同じ `Code.gs` を用意 → 新URL発行 → 各端末のURL設定を
  貼り替え → 1台で全件送信。
- 各端末がローカルに全データの控えを持つため、空の新バックエンドへ丸ごと移せる。

## 成功基準（実態として満たしている点）

- 1台で入力した案件・請求書・機材が、他端末で起動時に反映される。
- 売上表・部門別売上が全社の入力を合算して表示される。
- 案件・請求書の削除が他端末に伝播し、原則復活しない（請求書はLWWで保証）。
- 別々の案件・別々の機材の同時編集でデータが消えない。
- 同期オフ（URL未設定）でも従来どおり動作する。
- URLを貼り替えるだけでバックエンド（＝アカウント）を移行できる。
- ネット不調でもアプリが止まらない（未送信は後で再送）。
